# llama.cpp gfx1103 / Radeon 780M — MMVQ `nwarps=1` decode fix (+22–36% tg)

**Fixes slow token-generation (decode) on the AMD Radeon 780M (gfx1103, Phoenix/Hawk Point iGPU)
under ROCm/HIP llama.cpp.** A 1-file, gfx1103-only change to `ggml/src/ggml-cuda/mmvq.cu` that forces
`nwarps=1` for the decode (`mul_mat_vec_q`) path on this 12‑CU iGPU. Measured **+22–36% tg128** on
Gemma‑4‑26B‑A4B. Other GPUs untouched; prefill untouched; `test-backend-ops` passes.

**Branch:** `gfx1103-mmvq-nwarps` on this fork · **Patch:** `gfx1103-mmvq-nwarps.patch`
**Author:** Reuven Neumann. AI tools (Opus/Codex) assisted with investigation, benchmarking and drudgery;
code authored, built, tested and reviewed by me.

## Why it's slow (the problem)
Upstream commit `617db24` / PR #19478 raised RDNA3 MMVQ `nwarps` from 2→8 for `bs=1` decode — good for
big discrete RDNA3 GPUs (W7900, 96 CUs), but a **~30% decode regression on the 12‑CU 780M iGPU**: the extra
warps can't be used and add `__syncthreads()` / LDS-reduction overhead. (Same root cause as the open general
fix PR #20831; this is a simpler, gfx1103‑specific alternative.)

## The fix
gfx1103 gets its own MMVQ parameter table that **inherits RDNA3_0 unchanged and only overrides `nwarps`→1**
for the decode quants (`ncols_dst == 1`) where the regression was measured. Everything else (prefill, other
quants, other GPUs) is byte‑identical to upstream.

## Measured (current master, gfx1103/Radeon 780M, Gemma‑4‑26B‑A4B MoE, `tg128`, `-fa 1 -ngl 99`, r=3)
| quant | stock | PR #20831 (dynamic) | this patch (nwarps=1) |
|---|---:|---:|---:|
| Q4_K_M | 17.77 | 21.76 (+22%) | **22.80 (+28%)** |
| Q6_K | 15.66 | 17.59 (+12%) | **19.09 (+22%)** |
| Q8_0 | 12.81 | 14.48 (+13%) | **17.40 (+36%)** |

Prefill (`pp512`) is within noise (decode‑only change). Correctness: `test-backend-ops -o MUL_MAT` and
`-o MUL_MAT_ID` pass on gfx1103. Stability: 24h soak runs (Q8_0, Q6_K on Gemma‑4‑26B). Verified on bare‑metal
Ubuntu 24.04 and in a Proxmox LXC with the iGPU passed through.


**Option C — the change itself (one file, `ggml/src/ggml-cuda/mmvq.cu`):**
```diff
 enum mmvq_parameter_table_id {
     MMVQ_PARAMETERS_RDNA3_0,
+    MMVQ_PARAMETERS_RDNA3_0_GFX1103,
     MMVQ_PARAMETERS_RDNA4
 };

 static constexpr __device__ mmvq_parameter_table_id get_device_table_id() {
 #if defined(RDNA4)
     return MMVQ_PARAMETERS_RDNA4;
+#elif defined(__gfx1103__)
+    return MMVQ_PARAMETERS_RDNA3_0_GFX1103; // select before RDNA3_0
 #elif defined(RDNA3_0)
     return MMVQ_PARAMETERS_RDNA3_0;
 ...

 static __host__ mmvq_parameter_table_id get_device_table_id(int cc) {
     if (GGML_CUDA_CC_IS_RDNA4(cc)) { return MMVQ_PARAMETERS_RDNA4; }
+    if (cc == GGML_CUDA_CC_OFFSET_AMD + 0x1103) {
+        return MMVQ_PARAMETERS_RDNA3_0_GFX1103;
+    }
     if (GGML_CUDA_CC_IS_RDNA3_0(cc)) { return MMVQ_PARAMETERS_RDNA3_0; }
 ...

 // inside calc_nwarps(ggml_type type, int ncols_dst, mmvq_parameter_table_id table_id):
+    if (table_id == MMVQ_PARAMETERS_RDNA3_0_GFX1103) {
+        if (ncols_dst == 1) {
+            switch (type) {
+                case GGML_TYPE_Q4_K:
+                case GGML_TYPE_Q5_0:
+                case GGML_TYPE_Q5_K:
+                case GGML_TYPE_Q6_K:
+                case GGML_TYPE_Q8_0:
+                    return 1;
+                default: break;
+            }
+        }
+        return calc_nwarps(type, ncols_dst, MMVQ_PARAMETERS_RDNA3_0);
+    }
```

## Notes
- gfx1103‑only (`__gfx1103__` / `cc == GGML_CUDA_CC_OFFSET_AMD + 0x1103`); all other RDNA3_0 targets
  (gfx1100/1101/1102, W7900/W7800, RX 7000 dGPUs) delegate to the existing table unchanged.
- Q4_K_M / Q6_K / Q8_0 were benchmarked; Q5_0 / Q5_K are included as the same dispatch class (not separately
  benchmarked).
- Upstream status: not a merged PR. PR #20831 is the general (width-based) fix for the same regression; this
  is the narrower gfx1103‑specific lever. Data shared on the #20831 thread.
