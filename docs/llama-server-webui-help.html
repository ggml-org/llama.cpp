Sampling · 采样参数

⚠️ 核心页面！ 采样参数直接决定模型输出的质量、创造性和稳定性。本地 12B 模型尤其需要精心调节。
参数 默认 说明与取值影响 推荐值
Temperature 1 控制随机性的核心参数。
0：完全确定性输出，每次结果相同（适合代码生成、数据提取）
0.1~0.3：高度确定性，输出稳定可靠
0.5~0.7：轻微随机，适合正常对话
0.8~1.0：标准输出，有一定创造性
1.0~1.5：高创造性，但可能输出不稳定
>1.5：天马行空，容易胡说八道
典型场景：
  • 写代码/数学推理 → 0.1~0.3
  • 工具调用/Agent 循环 → 0.2~0.5
  • 日常聊天 → 0.6~0.8
  • 创意写作/头脑风暴 → 1.0~1.2
 稳定模式: 0.3
聊天模式: 0.7
创意模式: 1.1
Top K 64 候选 token 池大小。模型只从概率最高的 K 个 token 里选。
1：只看概率最高的 token（极度确定，适合翻译）
10~20：小候选池，输出很保守
40~60：标准范围
80~120：大候选池，输出更多样
0：关闭 Top K 采样（全选，不推荐） 		30~50
配合 Temperature 一起调
Top P 0.95 核采样阈值。从累积概率达到 P 的最小 token 集合中采样。比 Top K 更智能——当模型对某个词极度自信时（概率 0.99），就直接选它，不凑候选池。
0.5：只看前几个高概率 token，输出很保守
0.8~0.9：中等范围，适合工具调用
0.92~0.98：标准范围
1.0：不限制（等效于只看 Temperature）
Top K vs Top P：选哪个？
一般只用一个，推荐用 Top P（更智能适应）。如果要同时用，先跑 Top K 再跑 Top P，即 samplers 顺序中 top_k 应在 top_p 前面。
0.9
配合工具调用时收紧到 0.85
Min P 0.05 过滤掉概率低于"最高概率 token × Min P"的 token。防止模型偶尔抽风选到极低概率的词。
0：不过滤（可能偶尔抽风）
0.02~0.05：轻微过滤，标准设置
0.1~0.2：较强制过滤，输出更保守 		保持 0.05
Typical P （空） 按 token log-prob 与信息熵的差值排序筛选。这是一种更高级的采样方式。
设为空 = 不启用。一般不需要碰。 		不填，不启用
XTC probability 0 实验性功能。XTC（eXclude Top Choices）采样器：以一定概率剪掉最可能的 token，强制模型探索次优选择。
0：关闭（推荐）
0.3~0.5：中等强度探索
0.7~1.0：强力探索，输出会很 wild 		保持 0，实验性功能不稳定
XTC threshold 0.1 XTC 裁剪的阈值。只有概率低于此值的 token 会被考虑剪掉。 同上，不用动
Dynamic temperature range 0 动态温度范围。根据当前生成内容的"熵"自动调节温度。
0：关闭动态温度
0.2~0.5：轻微动态调节
一般不需要开，手动调 Temperature 就够了。 		保持 0
Dynamic temperature exponent 1 动态温度的平滑指数。控制温度变化的陡峭程度。 保持默认
Max tokens -1 单次回复最大 token 数。 -1 = 无限制，直到模型自己停下或上下文耗尽。
-1：⚠️ 本地模型高危！可能一直生成到上下文溢出或超时崩溃
512：极短回复
1024~2048：中等长度
4096~8192：长文本/代码生成
16384：超长输出（不推荐本地模型）
为什么不能设 -1？
本地模型缺乏云端 API 的全套安全截断机制。设 -1 时，如果模型"不知道停"（尤其是 Penalities 没调好的情况下），会一直狂生成直到上下文溢出或推理超时，然后整个 llama-server 崩溃退出。你昨天的崩溃就是这个原因之一。
一定不能设 -1！
代码/工具: 8192
聊天: 4096
小任务: 2048
Samplers （长链） 采样器的执行顺序链。默认顺序是经过验证的最优顺序：
penalties → dry → top_n_sigma → top_k → typ_p → top_p → min_p → xtc → temperature
不要随便改顺序！ 乱序可能导致参数互相干扰。 		不要改！
Backend sampling ❌ OFF 让采样器在 GPU/CUDA 后端上运行，加速采样过程。有 CUDA 加速时可开启。 有 GPU 建议开启

Penalties · 惩罚参数

⚠️ 防复读的核心页面！ 本地模型在长文本生成时极易陷入死循环复读。惩罚参数就是刹车系统。
参数 默认 说明与取值影响 推荐值
Repeat last N 64 回看最近 N 个 token 来判断重复。只影响 Repeat penalty 的作用范围。
32~64：标准范围，适合短对话
128~256：大窗口，能检测更远的重复（但计算量大）
-1：回看全部上下文 		64~128
Repeat penalty 1 最常用的防复读参数。 对已出现过的 token 施加惩罚，降低其被再次选中的概率。
1.0：不惩罚（关闭）
1.05~1.15：轻微惩罚，适合正常对话
1.15~1.3：中等惩罚，适合长文本生成
1.3~1.5：强力惩罚，适合极度复读场景，但可能导致输出不自然
>1.5：过度惩罚，模型会刻意回避常用词，输出怪异
副作用： Repeat penalty 太高会导致模型回避正常高频词（如"的"、"是"），输出变得不自然甚至语法错乱。从 1.1 开始试，逐步微调。
1.1~1.2
本地模型建议 1.15
Presence penalty 0 对已出现过的 token 直接扣分，不管出现了几次。鼓励模型使用新词。
0：关闭
0.1~0.3：轻微鼓励新词
0.5~1.0：强力鼓励新词，但可能变得啰嗦绕圈
>1.0：过度偏执于用新词
Presence vs Frequency：
  • Presence penalty：出现过的 token 统一罚一次
  • Frequency penalty：按出现次数累计惩罚
  • 一般用 Repeat penalty 就够了，这两个保持 0
 保持 0
Frequency penalty 0 按 token 出现次数累积惩罚。出现越多罚越狠。
0：关闭
0.1~0.3：轻微
0.5~1.0：中等，强制词汇多样性 		保持 0
DRY multiplier 0 🆕 智能防复读大杀器！ DRY (Don't Repeat Yourself) 是 llama.cpp 的特色功能，能检测语义级重复（而不仅仅是 token 级重复）。
0：关闭 DRY
0.2~0.5：轻微阻止重复
0.6~1.0：标准推荐范围
1.0~1.5：强力阻止重复
>1.5：可能过度限制，输出变短
为什么 DRY 比 Repeat penalty 更智能？
Repeat penalty 是机械地惩罚每个见过的 token；DRY 会检测"语义片段"的重复——比如模型一直在说类似的句式、结构，即使 token 不完全一样，DRY 也能识别并打断。对长文本/代码生成场景，DRY 是唯一可靠的反复读机制。
强烈建议开启！
0.6~1.0
本地模型入门：0.8
DRY base 1.75 DRY 惩罚的基数。控制 DRY 惩罚的基准强度。 保持 1.75
DRY allowed length 2 允许重复的最短片段长度。只有长度 >= 此值的连续片段才算重复。
1：连单字重复都检测（太敏感）
2：标准设置
3~5：只检测较长片段的重复 		保持 2
DRY penalty last N -1 DRY 的作用范围（回看 token 数）。
-1：使用全部上下文（推荐）
512~2048：限制回看范围，降低计算量 		保持 -1

📋 防复读方案对比

短对话 (<2K) Repeat penalty 1.1 就够了
长文本/代码 Repeat penalty 1.15 + DRY 0.8
严重复读场景 Repeat penalty 1.2 + DRY 1.0
高创造性（可接受偶尔重复） Repeat penalty 1.05 + DRY 0.3

Agentic · 智能代理循环

参数 默认值 说明与取值影响 推荐
Agentic turns 10 工具执行的最大循环次数。 达到上限后强制停止 Agent 循环。
1~2：只允许一轮工具调用（极度保守）
3~5：允许 2-3 轮工具调用，适合大多数场景
6~8：适合复杂多步任务
10：默认值
>10：可能陷入无限循环，浪费大量时间
对本地模型的影响：
每多一轮 = 多一次完整的推理请求。12B 模型每次推理 60~150 秒，10 轮最坏情况 = 25 分钟！实际使用中 80% 的任务 2-3 轮就完成了。
3~5
本地模型建议 3
Max lines per tool preview 25 工具输出在对话记录中保留的最后 N 行。中间内容会被丢弃以节省上下文。
10：只保留最后 10 行（非常省上下文）
25：标准设置
50~100：保留更多细节（更耗上下文）
0：不保留工具输出预览 		25
本地模型上下文紧张，不宜过大

Developer · 开发者选项

设置 默认值 说明与取值影响 推荐
Pre-fill KV cache after response ❌ OFF 免费的性能加速器！ 每次回复完成后，后台自动把对话重新提交给服务端做 KV cache 预填充。下次请求时 prompt 已经编码好了，首 token 延迟大幅降低。
效果： 省掉每次 5~15 秒的首 token 延迟。对多轮对话体验提升显著。
代价： 后台会占用一点 CPU/GPU 做预计算。对长上下文对话尤其有效。
⚠️ 已知问题： 在某些 llama.cpp 版本中，此功能可能触发 KV cache 检查点恢复问题（你昨天的崩溃根源）。如果开启后遇到崩溃，先关掉这个排查。
建议开启
有崩溃风险时先关闭排查
Disable reasoning content parsing ❌ OFF 关闭推理内容解析，让 thinking tokens 以内联方式返回而不是提取到单独字段。 保持关闭
Exclude reasoning from context ❌ OFF 发送消息前剥离历史中的 thinking 内容，不让模型看到自己之前的思考链。 保持关闭
Enable raw output toggle ❌ OFF 显示一个切换按钮，可以在 Markdown 和纯文本之间切换显示。 调试时有用
Custom JSON 空 自定义 JSON 参数，会原样发送给 API。
示例：
{"seed": 42, "stop": ["\n\n\n"], "logprobs": true}
可以覆盖任何 UI 未暴露的参数。 		高级用户按需使用
💡 Custom JSON 常用参数：
  • "seed": 42 — 固定随机种子，让输出可复现（调试用）
  • "stop": ["\n\n\n"] — 遇到三个连续换行就停止（防止无限生成）
  • "mirostat": 2, "mirostat_tau": 5 — 启用 Mirostat v2 采样器（替代 Temperature/Top-P 的另一种方案）
  • "logprobs": true, "top_logprobs": 3 — 返回 top-3 token 的概率分布（调试用）

MCP · 模型上下文协议

参数 默认值 说明与取值影响 推荐
Request timeout (seconds) 300 单次 MCP 工具调用的超时时间（秒）。
30~60：快速超时，适合本地 MCP 服务
120~300：标准设置，适合网络 MCP
600+：长超时，适合需要大量计算的服务
可按每个 MCP 服务器单独覆盖。 		300
外部搜索服务建议 60~120
📡 关于 MCP 的说明：
MCP（Model Context Protocol）允许你通过 --ui-mcp-proxy 启动参数让 llama-server 代理外部 MCP 服务，扩展模型的能力（如网络搜索、数据库查询等）。你之前用 --ui-mcp-proxy 连 anysearch.com 但连不上（日志里的 Failed to read connection 错误），是网络问题，不是参数问题。

Export Settings 导出所有 WebUI 设置为 JSON 文件。 适合备份调好的参数配置。
Import Settings 从 JSON 文件导入 WebUI 设置。 覆盖现有设置。建议先导出当前设置做备份。
⚠️ 所有设置存储在浏览器的 localStorage 中 — 清除浏览器数据会丢失所有设置！建议定期 Export Settings 备份。
