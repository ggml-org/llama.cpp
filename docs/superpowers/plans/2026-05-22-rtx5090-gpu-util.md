# Two-Stream Logits Readback Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Push RTX 5090 from 50% to >80% GPU utilization during single-request token generation by overlapping compute and logits readback on separate CUDA streams.

**Architecture:** Add a dedicated `readback_stream` to the CUDA backend context. Route `get_tensor_async` and `get_tensor_2d_async` onto this stream with event-based dependency on the compute stream. Make `synchronize` wait on the readback stream (which transitively covers compute via the event chain). The GPU copy engine handles the memcpy while SMs are free for the next forward pass.

**Tech Stack:** CUDA C++, CMake, ggml backend API

**Prerequisites:** The output buffer is already allocated as pinned memory (`ggml_backend_cuda_host_buffer_type` → `cudaMallocHost`), so `cudaMemcpyAsync` to host is already asynchronous.

### Task 2: Route get_tensor_async onto readback stream

**Files:**
- Modify: `ggml/src/ggml-cuda/ggml-cuda.cu:3175-3182` — `ggml_backend_cuda_get_tensor_async`
- Modify: `ggml/src/ggml-cuda/ggml-cuda.cu:3195-3204` — `ggml_backend_cuda_get_tensor_2d_async`

**Rationale:** Redirect device-to-host async copies from the compute stream to the readback stream, with event-based dependency so the copy only starts after compute kernels finish.

- [ ] **Step 1: Replace `ggml_backend_cuda_get_tensor_async`**

In `ggml/src/ggml-cuda/ggml-cuda.cu` (line ~3175), replace the function body:

```cpp
static void ggml_backend_cuda_get_tensor_async(ggml_backend_t backend, const ggml_tensor * tensor, void * data, size_t offset, size_t size) {
    ggml_backend_cuda_context * cuda_ctx = (ggml_backend_cuda_context *) backend->context;
    ggml_backend_buffer_t buf = tensor->view_src ? tensor->view_src->buffer : tensor->buffer;

    GGML_ASSERT(buf->buft == ggml_backend_cuda_buffer_type(cuda_ctx->device) && "unsupported buffer type");

    cudaStream_t rb_stream = cuda_ctx->readback_stream_get();

    // Record event on compute stream marking end of kernels
    CUDA_CHECK(cudaEventRecord(cuda_ctx->readback_event, cuda_ctx->stream()));
    // Readback stream waits for compute to finish
    CUDA_CHECK(cudaStreamWaitEvent(rb_stream, cuda_ctx->readback_event, 0));
    // Launch async copy on readback stream (pinned host memory already guaranteed by cuda_host buffer type)
    CUDA_CHECK(cudaMemcpyAsync(data, (const char *) tensor->data + offset, size, cudaMemcpyDeviceToHost, rb_stream));
    // Record completion event so synchronize can wait on it
    CUDA_CHECK(cudaEventRecord(cuda_ctx->copy_done_event, rb_stream));
}
```

### Task 3: Update synchronize to wait on readback stream

**Files:**
- Modify: `ggml/src/ggml-cuda/ggml-cuda.cu:3364-3377` — `ggml_backend_cuda_synchronize`

**Rationale:** When `synchronize` is called (e.g., by `llama_synchronize` after `get_tensor_async`), it must wait for the readback stream's copy to complete. Waiting on the readback stream transitively covers the compute stream because the readback waits on the compute's event.

- [ ] **Step 1: Replace `ggml_backend_cuda_synchronize`**

In `ggml/src/ggml-cuda/ggml-cuda.cu` (line ~3364), replace the entire function:

```cpp
static void ggml_backend_cuda_synchronize(ggml_backend_t backend) {
    ggml_backend_cuda_context * cuda_ctx = (ggml_backend_cuda_context *)backend->context;

    // Synchronize compute stream — covers all kernels and cuBLAS work
    if (!ggml_cuda_adaptive_wait(cuda_ctx->stream(), cuda_ctx->device,
            &cuda_ctx->t_wait_us, &cuda_ctx->n_wait_ops,
            &cuda_ctx->t_wait_phase1_us, &cuda_ctx->t_wait_phase2_us,
            &cuda_ctx->t_wait_phase3_us,
            &cuda_ctx->t_wait_fallback_us, &cuda_ctx->n_wait_fallback)) {
        CUDA_CHECK(cudaStreamSynchronize(cuda_ctx->stream()));
    }

    // Synchronize readback stream — covers in-flight device-to-host copies.
    // The copy_done_event is recorded on the readback stream after each
    // memcpyAsync, so synchronizing it guarantees the copy finished.
    if (cuda_ctx->copy_done_event != nullptr) {
        cudaError_t err = cudaEventSynchronize(cuda_ctx->copy_done_event);
        if (err != cudaSuccess) {
            CUDA_CHECK(cudaStreamSynchronize(cuda_ctx->readback_stream));
        }
    }

    GGML_UNUSED(backend);
}
```

**Type consistency:**

- `readback_event` / `copy_done_event` type: `cudaEvent_t` — used consistently with cudaEventRecord/Query/Synchronize

- Destructor cleanup order: readback_stream → events → safe because stream is synchronized before destroy

**Correctness check:**
- Event dependency chain: `compute_stream → readback_event → readback_stream (cudaMemcpyAsync) → copy_done_event`
- `synchronize()` waits on: compute stream (adaptive wait) + copy_done_event
- Multiple sequential `get_tensor_async` calls: each records a new event on compute, readback stream chains via `cudaStreamWaitEvent` before each memcpy — they execute sequentially on the readback stream in order, which is correct
- The `copy_done_event` is reused across calls. Each new `cudaEventRecord(copy_done_event, rb_stream)` overwrites the previous. The synchronize waits on the latest record. This is correct because the readback stream is sequential — if the latest memcpy finished, all previous ones on the same stream also finished.
