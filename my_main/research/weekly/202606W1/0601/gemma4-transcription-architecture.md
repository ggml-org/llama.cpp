# Gemma 4 を使った文字起こしアプリの整理

## 結論

Gemma 4 を使ってアプリ内で文字起こしをやる場合、主な実装責務はアプリ側にあります。

llama.cpp 側は、Gemma 4 をロードして音声入力を受け付ける推論基盤として使えますが、リアルタイム文字起こし製品として必要な処理全体を肩代わりしてくれるわけではありません。

特に重要なのは、現在の構成では「マイク音声をそのまま連続ストリームで投げるだけで高品質なリアルタイム ASR が完成する」という形ではないことです。実際には、アプリ側で音声を扱いやすい単位に分け、送信し、結果をつなぎ、表示を安定させる責務を持つ前提で設計した方がよいです。

## まず押さえるべき前提

Gemma 4 は、llama.cpp 上で音声入力を扱えるマルチモーダルモデルとして動かせる余地があります。

ただし、音声はこのリポジトリでも実験的な扱いです。したがって、Gemma 4 を使う場合は「音声理解ができる LLM を利用して、文字起こし体験を組み立てる」という考え方が近く、専用 ASR エンジンをそのまま置き換える感覚では見ない方が安全です。

つまり、Gemma 4 でやるなら次の期待値調整が必要です。

- 精度は入力条件に強く依存する
- 長時間連続音声ではアプリ側の分割設計が重要になる
- 遅延はモデルサイズ、量子化、GPU 条件、チャンク長に大きく左右される
- 低遅延な逐次字幕は、モデル単体ではなくアプリの制御設計で作る

## 責務の切り分け

### llama.cpp 側の責務

llama.cpp 側は、基本的に次の役割に留まります。

- Gemma 4 モデルと mmproj をロードする
- 音声入力を受け取れる推論サーバを立てる
- 音声を含むリクエストに対して推論結果を返す
- ストリーミング応答や性能監視の土台を提供する

この意味では、「こっちの LLM 基盤側でやることはあまりないのか」という問いには、かなり Yes に近いです。

ただし、完全にゼロではありません。基盤側では少なくとも次を整えておく必要があります。

- Gemma 4 の音声対応モデルを選ぶ
- そのモデルに対応する mmproj を揃える
- llama-server を安定して起動できるようにする
- 1 リクエストあたりの応答時間、同時接続数、メモリ使用量を把握する
- アプリから叩く API の入出力契約を固定する

つまり、基盤側の仕事は「音声理解モデルを安定して呼べる推論サービスにすること」です。

### アプリ側の責務

アプリ側の責務はかなり多いです。リアルタイム文字起こし体験の品質は、むしろこちらで決まります。

主に必要なのは次の設計です。

- マイク入力の取得
- 音声チャンク化
- 無音検出または発話区間検出
- 送信タイミング制御
- 部分結果と確定結果の管理
- 重複テキストの除去とマージ
- 遅延と精度のトレードオフ調整
- 通信失敗時の再送や復旧
- UI 上の字幕表示と状態遷移管理

Gemma 4 を使う場合、アプリ側で実装するべきことをさらに具体化すると次の通りです。

## アプリ側で実装するべきこと

### 1. 音声を連続ストリームではなく短い単位に区切る

Gemma 4 を使ったリアルタイム文字起こしでは、アプリがマイク音声を短いチャンクに分けて送る設計が基本になります。

考え方としては、数百ミリ秒から数秒程度の区間ごとに音声を切り出し、順番に推論へ渡します。

ここでの目的は次です。

- 待ち時間を短くする
- モデルに長すぎる音声を一度に投げない
- UI に途中結果を出せるようにする
- 失敗時に小さい単位で再送できるようにする

リアルタイム性は、このチャンク長の設計に強く依存します。短いほど反応は速くなりやすいですが、文脈不足で精度が落ちやすくなります。長いほど精度は安定しやすいですが、遅延が大きくなります。

### 2. 無音検出を入れる

常に一定間隔で送るだけだと、無音や雑音まで推論にかけ続けることになります。

そのため、アプリ側で VAD か簡易な無音判定を入れて、発話がある時だけ送る構成にした方がよいです。

これにより次の効果があります。

- 無駄な推論回数が減る
- コストとレイテンシが下がる
- 字幕のちらつきが減る
- 無音部分が誤認識されにくくなる

Gemma 4 自体にこの制御を任せるのではなく、入力前にアプリ側で整理するのが本筋です。

### 3. チャンク間に少し重なりを持たせる

音声をきれいに区切るのは難しいため、チャンク境界で単語が切れることがあります。

これを避けるため、前後のチャンクに少しオーバーラップを持たせる設計が有効です。

ただし、オーバーラップを入れると同じ内容が重複して返ってくることがあるため、アプリ側で重複除去や差分マージが必要になります。

つまり、リアルタイム文字起こしでは次の 2 つはセットです。

- 境界欠落を防ぐための重なり
- 重複表示を防ぐためのマージ処理

### 4. 部分結果と確定結果を分けて扱う

UI では、いま話している途中の暫定字幕と、話し終えて確定した字幕を分けて管理した方がよいです。

この設計を入れないと、モデルの返答が少し変わるたびに全文字列を書き換えることになり、見た目が不安定になります。

考え方としては次です。

- 進行中の発話は暫定表示する
- 区切りが確実になった部分だけ確定表示へ移す
- 次チャンクで文言が変わる可能性がある範囲は確定しない

この責務はアプリ側にあります。推論基盤に期待するより、表示レイヤで制御する方が自然です。

### 5. Gemini 的な全文再生成ではなく差分マージを考える

Gemma 4 による音声理解では、返ってくるテキストが毎回完全に安定するとは限りません。

そのため、チャンクごとの出力を毎回丸ごと置き換えるより、前回結果との差分を見ながらつなぐ考え方が必要です。

重要なのは次です。

- 前回の末尾と今回の先頭が重複していないかを見る
- 少しだけ言い換わった部分を許容する
- 確定済み区間はなるべく再編集しない
- 一時的な誤認識で UI が大きく揺れないようにする

リアルタイム字幕の品質は、モデルの絶対精度だけでなく、この差分統合のうまさでかなり変わります。

### 6. プロンプトを文字起こし向けに固定する

Gemma 4 は汎用 LLM 寄りなので、アプリ側では「何を返してほしいか」をかなり明示した方が安定します。

概念としては、毎回のリクエストで次の方針を固定した方がよいです。

- 要約ではなく逐語的な文字起こしを優先する
- 聞き取れない部分は推測しすぎない
- 余計な説明文をつけない
- 出力形式を一定にする
- 必要なら言語を明示する

Gemma 4 でやる以上、「音声を聞いて自由回答する」のではなく、「音声から転写テキストを返すモードに寄せる」ためのアプリ側制御が必要です。

### 7. 長時間セッションの扱いを決める

会議や連続対話のような長時間セッションでは、アプリがどこまで過去文脈を持つかを決める必要があります。

文字起こし用途では、過去全文を常に再入力するより、次のように分けて考えた方がよいです。

- 音声認識そのものは短い近傍文脈で回す
- UI 表示用の履歴は別管理する
- 必要な場合だけ話者情報や直前文脈を補助として渡す

Gemma 4 に全履歴を背負わせる設計は、遅延もコストも増えやすいです。

### 8. 音声フォーマット変換をアプリ側で吸収する

実運用では、ブラウザやモバイル環境によって録音形式がぶれます。

したがって、アプリ側でサーバが受けやすい形式に正規化する層を持つべきです。

概念的には次を吸収します。

- 入力デバイスごとのサンプルレート差
- 録音形式の差
- モノラル化や正規化の必要性
- 送信サイズの制御

これは推論基盤ではなく、クライアントまたはアプリのメディア層の責務です。

## Gemma 4 を使う時の現実的な見方

Gemma 4 を使いたいという前提なら、設計思想としては「Gemma 4 を ASR 専用機とみなす」のではなく、「Gemma 4 の音声理解能力を、アプリ側の前処理と後処理で文字起こし用途に寄せる」と考えるのが妥当です。

つまり、Gemma 4 を選ぶなら重要なのは次です。

- モデル選定より、音声分割と結果統合の設計
- 単発精度より、連続利用時の表示安定性
- API の機能より、アプリの状態管理
- 推論の賢さより、入力の整え方

専用 ASR モデルより苦労する可能性はありますが、Gemma 4 に統一したいのであれば、その差はアプリ設計で吸収する方針になります。

## この llama.cpp 側で最低限やること

このリポジトリ側でやることは、基本的には少なめです。優先順位としては次です。

1. Gemma 4 の音声入力対応モデルを確定する
2. mmproj を含めた起動構成を固定する
3. API の入力形式と返却形式を決める
4. 1 チャンクあたりの平均遅延を測る
5. 実用上の同時接続数を把握する

ここでいう基盤側のゴールは、賢いアプリを作ることではありません。アプリが安心して呼べる、安定した音声推論サービスを用意することです。

## 実装方針としておすすめの分け方

もし別リポジトリのアプリで開発するなら、責務分離は次の形が分かりやすいです。

### アプリ層

- 録音
- VAD
- チャンク分割
- 送信制御
- 再送制御
- 部分字幕と確定字幕の管理
- 差分マージ
- UX 制御

### 推論基盤層

- Gemma 4 のロード
- 音声入力 API の公開
- ストリーミング応答
- 性能監視
- モデル更新管理

### 運用層

- ログ収集
- レイテンシ監視
- モデル切替判断
- 精度評価
- 障害時のフォールバック判断

## 最後の判断

Gemma 4 前提で進めるなら、アプリ側でやるべきことは多いです。逆に、llama.cpp 側は音声推論サーバとして安定稼働できれば十分で、文字起こし体験の本質はアプリ側で作ると考えてよいです。

要するに、今回のテーマでは次の理解が一番近いです。

- llama.cpp 側は基盤
- Gemma 4 は推論エンジン
- リアルタイム文字起こし体験はアプリ側の設計で成立する

そのため、「こっちの LLM 基盤の方ではやることはあまりないのかな」という認識は概ね正しいです。ただし、少ない代わりに基盤側ではモデル選定、起動構成、API 契約、遅延把握だけは先に固めておくべきです。
